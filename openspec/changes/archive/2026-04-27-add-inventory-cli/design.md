## Context

CodeXray의 첫 변경. `constraints.md`에 따르면 스택은 vibe-coding 단계에서 결정한다. 인벤토리 CLI는 이후 의존성 그래프·정량 평가·핫스팟 매트릭스가 모두 재사용할 입력 파이프라인(파일 워킹, `.gitignore` 처리, 언어 분류)을 깔기 때문에, 이 시점의 스택·라이브러리 선택이 후속 모듈에 그대로 전파된다. 검증 대상은 사용자의 작은 게임 프로젝트(~수백~수천 파일 규모로 가정).

## Goals / Non-Goals

**Goals:**
- 후속 분석 모듈이 재사용할 결정론적 입력 파이프라인의 기초(파일 열거 + 무시 규칙 + 언어 분류)를 한 번 만들고 끝내는 것
- 단일 명령 한 줄로 표 출력. 외부 네트워크·AI 호출·설정 파일 없음
- 5초 내 완료(검증 프로젝트 기준)

**Non-Goals:**
- 의존성 그래프 / 호출 관계 / 진입점 식별
- AI 평가, JSON·Markdown 리포트 출력
- Git URL · 압축 파일 입력
- 멀티 언어 통계 외 정량 지표(복잡도·중복도 등)
- 설정 파일·플러그인 시스템

## Decisions

### Decision 1: 구현 언어 — Python 3.11+
**선택 이유:**
- MVP 대상 언어에 Python이 포함 → dogfooding 자연스러움
- 파일 워킹·`.gitignore` 매칭·텍스트 파싱·표 출력 모두 표준 라이브러리 + 소수 검증된 라이브러리로 해결 (`pathlib`, `pathspec`, `rich`)
- AI 평가 단계에서 LLM SDK 생태계가 가장 두텁다 (`anthropic`, `openai`)

**대안과 기각 사유:**
- **Node/TypeScript**: 표 출력·CLI 라이브러리 좋지만 파일 시스템 처리에서 Python 대비 이점 없음, 후속 AI 통합도 동급
- **Go**: 단일 바이너리 배포 매력적이나 LoC 카운트·언어 분류 정도에 정적 타입과 컴파일 비용은 과함. 특히 빠른 vibe 반복에 부적합
- **Rust**: 성능 무관, 1차 MVP 단계에서 학습·반복 비용 너무 큼

### Decision 2: 패키지 매니저 — uv
**선택 이유:** 단일 lockfile, 가상환경·의존성 설치를 한 명령으로 처리, `pyproject.toml` 표준 호환. 후속 단계에서 다중 도구를 추가할 때 환경 일관성 유지 비용이 가장 낮다.

**대안:** `pip + venv`(수동성 높음), `poetry`(과한 기능, 느림), `pdm`(생태계 작음).

### Decision 3: `.gitignore` 처리 — `pathspec` 라이브러리
**선택 이유:** Git 자체와 동일한 매칭 의미. 직접 구현 시 부정 패턴(`!`)·중첩 `.gitignore`에서 미묘한 버그 위험.

**기본 무시 디렉토리:** `pathspec`이 처리하는 `.gitignore`와 별개로 다음을 항상 제외 — `.git`, `node_modules`, `dist`, `build`, `venv`, `.venv`, `__pycache__`, `.next`, `target`. (프로젝트가 `.gitignore`에 안 적었더라도 분석 가치 0인 디렉토리.)

### Decision 4: 표 출력 — `rich.table`
**선택 이유:** 터미널 폭 자동 정렬·유니코드·박스 스타일 모두 무료. 표준 출력 파이프 시 자동으로 ANSI를 끈다 → 스크립트 친화. 추후 인터랙티브 대시보드 단계에서 `rich.console` 컴포넌트 재사용 가능성.

### Decision 5: 언어 분류 — 정적 매핑 테이블
1차 우선 언어 5개의 확장자 매핑만 코드에 둔다.

| 언어 | 확장자 |
|---|---|
| Python | `.py` |
| JavaScript | `.js`, `.jsx`, `.mjs`, `.cjs` |
| TypeScript | `.ts`, `.tsx` |
| Java | `.java` |
| C# | `.cs` |

**확장자가 매핑에 없는 파일은 집계에서 제외**(unknown 행을 만들지 않음). 이유: 첫 번째 표는 "이 코드베이스의 주력 언어가 무엇인가"에 답해야 하며, `.md`·`.json`·이미지 등이 섞이면 신호가 흐려진다. linguist 수준의 정밀 분류는 후속 변경의 몫.

### Decision 6: LoC 정의 — 비어있지 않은 줄 수
공백·주석을 가리지 않는다. 주석 인식은 언어별 lexer가 필요해 1차 범위 초과. "비어있지 않은 줄"은 결정론적이고, 첫 표의 신호로 충분하다. 진짜 SLOC가 필요해지면 정량 평가 단계에서 재정의.

### Decision 7: 출력 컬럼 — 4개 고정
`language`, `file_count`, `loc`, `last_modified_at`. 정렬은 `loc` 내림차순. `last_modified_at`은 해당 언어 파일들의 mtime 중 최댓값을 ISO-8601 로컬 타임존으로 표기.

### Decision 8: CLI 프레임워크 — `typer`
**선택 이유:** 데코레이터 기반, 타입 힌트로 자동 검증·도움말 생성. `argparse` 대비 코드 절반.

## Risks / Trade-offs

- **[리스크] 5초 예산 초과** → 파일 수가 수만에 달하는 모노레포에서 mtime/크기 측정이 누적 비용. **완화:** 초기 구현은 `os.scandir` + `Path.read_bytes` 기반 단일 패스, 멀티프로세싱은 도입하지 않는다. 검증 프로젝트가 5초 안에 들어오면 통과. 향후 큰 레포에서 막힐 때 재논의.
- **[리스크] 언어 분류 누락이 후속 단계 데이터 결손으로 전파** → 첫 변경에서 5개만 받기로 한 결정. **완화:** spec에 "확장자 매핑은 후속 변경에서 확장된다"를 명시하고, 매핑 테이블을 단일 모듈에 격리해 추가 비용을 최소화.
- **[리스크] `.gitignore`가 없는 프로젝트** → `pathspec` 매칭은 빈 규칙으로 동작. 기본 무시 디렉토리만 적용된다. 의도된 동작.
- **[리스크] 심볼릭 링크 순환** → `Path.walk(follow_symlinks=False)` 또는 동등 옵션으로 미리 차단.
- **[트레이드오프] LoC를 "비어있지 않은 줄 수"로 정의 → 주석 위주의 자동 생성 코드(예: protobuf 산출물)에서 과대 집계** 위험. 1차에서는 수용. 정량 평가 단계에서 SLOC로 교체.
