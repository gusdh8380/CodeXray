# INTENT — CodeXray
> lifecycle: build 진행 중

## Why

낯선 코드베이스를 마주치면 막막하다 — 개발자도, 비개발자도.
파일이 수십 개, 관계는 복잡하고, 어디서부터 봐야 할지 모른다.
AI에게 물어봐도 파편적이고 정리가 안 된다.

코드베이스가 스스로 자기를 설명해주면 어떨까.
레포 하나를 입력하면, 그 결과물이 "이 코드가 뭔지"를
처음 보는 사람에게도 명확하게 전달해야 한다.

쓸 수 있어야 하는 세 상황:
1. 다른 사람의 레포를 빠르게 파악·학습
2. 내 프로젝트에 대한 셀프 피드백
3. 비개발자(바이브코더)가 자신의 프로젝트를 파악하고 다른 사람에게 설명할 수 있다

## What

- [ ] 레포 경로/URL 하나를 입력하면 전체 분석 결과가 나온다
- [ ] 웹 화면 자체가 발표 자료로 쓸 수 있을 만큼 깔끔하다
      (PPT를 따로 만드는 게 아니라, 이 웹을 열어놓고 설명한다)
- [ ] 코드를 몰라도 프로젝트 상태와 다음 행동을 파악할 수 있다
- [ ] 전문 용어가 나오더라도 비개발자가 맥락을 파악할 수 있다
- [ ] 시니어 개발자도 "오, 이거 괜찮은데" 할 분석 깊이가 있다
      (근거 코드 인용, 설계 문제 지적, 수치 기반 평가)
- [ ] "다음에 뭘 해야 하나"가 결과에서 바로 나온다
- [ ] 어떤 언어로 된 레포든 분석할 수 있다
- [ ] 분석 결과의 질이 기다릴 가치가 있다
- [ ] 분석 중 대기 경험이 지겹지 않다 (로딩 UX)

## Not

- 코드를 자동으로 고치지 않는다 — 분석/평가만
- AI의 판단이 근거 없이 제시되지 않는다
  (규칙 기반 증거 위에서만 AI 해석이 나온다)
- 최종 결정은 사람이
- SaaS 없음 — 로컬 실행 우선
- AI 분석 시 코드가 외부 서버로 전송된다
  학습 데이터로는 사용되지 않지만, 기밀 코드에는 사용자가 직접 판단해야 한다

## Learnings

- **Intent drift 2회** (2026-04-28, 2026-04-29) — Why 없이 What 목록만 있으면
  AI가 목록 소화 기계가 된다. 매 세션 시작 시 이 문서 먼저.
- **"웹이 곧 발표 자료"** — PPT export가 아니라 웹 UX 품질 자체가 목표.
  현재 개발자도 읽기 어렵다 = 핵심 목표 미달 상태.
- **비개발자 청중이 Why의 핵심인데 Intent에 없었다** — Vision에만 있어서
  drift 발생. 처음부터 Why에 넣었어야 했다.
- **ROBOCO 자산 미활용** — `/opsx:propose` 먼저 호출하는 습관 필요.
- **자기 적용이 최고의 검증** — CodeXray로 CodeXray를 분석하면 실제 문제 발견.
- **하이브리드 AI 방향 확정** (2026-04-28) — 규칙이 증거 수집, AI가 해석.
  순수 규칙은 맥락이 없고, 순수 AI는 근거가 없다. 둘을 결합해야 신뢰할 수 있다.
  현재 하드코딩된 수치를 AI 해석의 입력 자료로 활용.
- **바이브코더는 코드 이해가 아니라 프로젝트 파악이 목표** — "이 코드가 뭔지"를
  설명할 수 있으면 충분하다. 코드 자체를 이해시키려 하면 방향이 틀어진다.
