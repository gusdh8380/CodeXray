## Context

`vibe-detection-rebalance` 가 starter_guide 를 제거한 후 *vibe coding 자체를 배우려는 사용자* 의 진입점이 도구 안에 없다. 옵션은 (A) 도구 안 어딘가에 다시 진입점, (B) README 외부 자료 링크, (C) 별도 학습 사이트. 이 중 가장 가벼운 B 채택 — 사용자가 도구를 *이미 깐* 시점이라 README 확인 가능성이 높고, 외부 권위 자료 인용이 자체 작성 가이드보다 신뢰도 높음.

## Goals / Non-Goals

**Goals:**
- README 에 한 섹션 추가, 자료 8–10 개 + 한 줄 요약.
- 평가 철학 토글 8 번 섹션 (출처 인용) 과 *동일한 자료 셋* 유지 — 사용자가 도구 안에서 본 출처와 README 가 일치.
- 회귀 차단을 위한 spec 요구사항 1 개.

**Non-Goals:**
- 자체 작성 vibe coding 튜토리얼 (외부 자료가 더 나음, 우리가 만들 시간 가성비 낮음).
- 별도 docs/vibe-coding-resources.md 분리 (지금 단계엔 README 한 섹션이 충분, 나중에 길어지면 분리).
- 도구 화면 안 진입점 (옵션 A' 결정 유지).

## Decisions

### 결정 1: README 한 섹션, 별도 파일 분리 안 함

README 가 이미 길지 않음. 한 섹션 추가가 전체 길이의 큰 비중 아님. 나중에 자료가 20+ 개로 늘면 그때 `docs/vibe-coding-resources.md` 분리.

**대안 — 즉시 docs/ 분리**: 가짜 미래 대비. YAGNI.

### 결정 2: 자료 8–10 개, 평가 철학 토글과 동일

토글 8 번 섹션이 이미 자료 정리해둠. 그대로 가져와 README 에 — 한 곳 수정으로 두 곳 동기화 가능. 같은 사람·자료 두 번 반복하지 않게 *이름 + 한 줄 요약 + 링크* 형식 통일.

**대안 — 더 많은 자료**: 자료 풀이 너무 많으면 사용자가 어디부터 봐야 할지 모름. 8–10 개가 *의미 있는 다양성* 의 적정선.

### 결정 3: 한 줄 요약 동봉, 단순 링크 덤프 회피

각 자료에 *왜 이걸 봐야 하는지* 한 줄. 예:
- "Anthropic — Claude Code Best Practices: CLAUDE.md 작성 원칙"
- "Karpathy — vibe coding 용어의 시작점: AI 와 작업할 때 인간 미감의 역할"

링크만 있으면 사용자가 *클릭 비용* 을 결정 못 함. 한 줄이면 우선순위 정해짐.

### 결정 4: spec ADDED 1 개로 회귀 차단

"README 에 vibe coding 학습 자료 셋이 유지된다" 를 vibe-coding-insights spec 에 ADDED 로 박음. 이 요구사항이 있으면:
- 미래 누군가 README 를 줄이려 할 때 spec 검사로 차단
- 자료 셋 변경 시 평가 철학 토글과 동기화해야 한다는 압력 발생

## Risks / Trade-offs

- [README 가 길어짐 → 사용자 첫 인상 부담] → 섹션 위치를 README 끝부분 ("학습 자료") 로. 도구 사용 흐름 (설치·실행·이해) 다음에 둠.
- [외부 링크 dead link] → archive 시 수동 점검 + spec 에 *링크 유효성 자체는 강제 안 함* (외부 의존).
- [자료 셋이 평가 철학 토글과 따로 갱신됨] → spec ADDED 요구사항이 *동기화 의무* 를 명시. PR 리뷰에서 토글 변경 시 README 도 보라는 트리거.
