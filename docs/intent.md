# INTENT — CodeXray
> lifecycle: build 진행 중

## Why

낯선 코드베이스 앞에서 가장 막막한 사람은 비개발자 바이브코더다.
자기 프로젝트인데도 코드를 못 읽으니 다음 행동을 결정 못 한다.
AI에게 물어봐도 답이 파편적이고, 어디부터 봐야 할지조차 모른다.

코드베이스가 스스로 자기를 설명해주면 어떨까.
레포 하나를 입력하면, 그 결과물이 "이 코드가 뭔지"를
비개발자에게도 명확하게 전달해야 한다.

CodeXray는 비개발자 바이브코더를 1순위로 돕는다.
개발자(셀프 피드백, 학습용)도 같은 도구를 쓸 수 있지만 부산물이다.
디자인 결정이 충돌하면 항상 비개발자가 이긴다.

쓸 수 있어야 하는 세 상황 (우선순위 순):
1. (1순위) 비개발자(바이브코더)가 자신의 프로젝트를 파악하고 다음 행동을 결정한다
2. (부수) 다른 사람의 레포를 빠르게 파악·학습
3. (부수) 내 프로젝트에 대한 셀프 피드백

부수 사용자는 1순위에서 갈라져 나오는 효과로 충족되면 OK,
별도로 챙기느라 1순위가 흐려지면 NO.

## What

- [ ] 레포 경로/URL 하나를 입력하면 전체 분석 결과가 나온다
- [ ] 웹 화면 자체가 발표 자료로 쓸 수 있을 만큼 깔끔하다
      (PPT를 따로 만드는 게 아니라, 이 웹을 열어놓고 설명한다)
- [ ] 브리핑 영역은 코드 몰라도 프로젝트 상태와 다음 행동을 파악할 수 있다
- [ ] 브리핑 영역의 톤은 비개발자 청자 100% — 메트릭 용어는 평어로 풀어 쓰고,
      AI 추천에는 검토 경고를 명시한다
- [ ] 다음 행동(Next Actions)은 비개발자가 다음 AI 세션에 그대로 들고 갈 수 있는
      형태다 — 이해→작업→검증 3단을 갖춘다
- [ ] 상세 분석 토글은 개발자도 "오, 이거 괜찮은데" 할 깊이
      (근거 코드 인용, 설계 문제 지적, 수치 기반 평가)
- [ ] 어떤 언어로 된 레포든 분석할 수 있다
- [ ] 분석 결과의 질이 기다릴 가치가 있다
- [ ] 분석 중 대기 경험이 지겹지 않다 (로딩 UX)

## Not

- 코드를 자동으로 고치지 않는다 — 분석/평가만
- AI의 판단이 근거 없이 제시되지 않는다
  (규칙 기반 증거 위에서만 AI 해석이 나온다)
- 최종 결정은 사람이
- SaaS 없음 — 로컬 실행 우선
- AI 분석 시 코드가 외부 서버로 전송된다
  학습 데이터로는 사용되지 않지만, 기밀 코드에는 사용자가 직접 판단해야 한다
- 브리핑 영역에서 시니어 만족도를 위해 메트릭 용어·전문 표현을 도입하지 않는다
  — 시니어가 깊이를 원하면 상세 분석 토글로 간다

## Learnings

- **Intent drift 2회** (2026-04-28, 2026-04-29) — Why 없이 What 목록만 있으면
  AI가 목록 소화 기계가 된다. 매 세션 시작 시 이 문서 먼저.
- **"웹이 곧 발표 자료"** — PPT export가 아니라 웹 UX 품질 자체가 목표.
  현재 개발자도 읽기 어렵다 = 핵심 목표 미달 상태.
- **비개발자 청중이 Why의 핵심인데 Intent에 없었다** — Vision에만 있어서
  drift 발생. 처음부터 Why에 넣었어야 했다. (2026-04-30 본문 반영 완료)
- **ROBOCO 자산 미활용** — `/opsx:propose` 먼저 호출하는 습관 필요.
- **자기 적용이 최고의 검증** — CodeXray로 CodeXray를 분석하면 실제 문제 발견.
- **하이브리드 AI 방향 확정** (2026-04-28) — 규칙이 증거 수집, AI가 해석.
  순수 규칙은 맥락이 없고, 순수 AI는 근거가 없다. 둘을 결합해야 신뢰할 수 있다.
  현재 하드코딩된 수치를 AI 해석의 입력 자료로 활용.
- **바이브코더는 코드 이해가 아니라 프로젝트 파악이 목표** — "이 코드가 뭔지"를
  설명할 수 있으면 충분하다. 코드 자체를 이해시키려 하면 방향이 틀어진다.
- **페르소나 분열 함정** (2026-04-30) — Intent에서 "개발자도, 비개발자도"를 동격으로
  두면 Next Actions 같은 처방 영역이 가운데로 흘러간다. 페르소나 우선순위는 Why에서
  못 박아야 한다. 위(브리핑) = 비개발자, 아래(상세 분석 토글) = 개발자로 영역
  분리하는 결정으로 해소.
