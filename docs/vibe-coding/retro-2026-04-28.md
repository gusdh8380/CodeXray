## 의외였던 인사이트

### A. type-resolution 정확도가 등급을 바꾼다
같은 코드베이스, 같은 측정 도구, 정확도만 다르게 → CivilSim **F(32) → D(40)**. namespace 1:N 매핑에 의한 거짓 결합도가 평균 fan_inout을 17.25에서 6.72로 부풀렸다. **측정 도구를 만들 때 정확도는 결과의 의미 자체를 바꾼다**. 1차 결과는 진실이 아니라 "지금 도구의 시야"임을 항상 의식.

### B. 프롬프트는 한 번에 동작한다 (조건이 맞으면)
claude·codex 둘 다 첫 시도에서 정확한 JSON 스키마로 응답. 비결: 프롬프트가 (1) 명확한 규칙, (2) JSON 스키마 예시, (3) "다른 텍스트 금지" 강제, (4) 한국어로 일관. 4박자가 맞으면 retry·fallback 없이 동작.

### C. Markdown 리포트가 JSON보다 사용자에게 가치 있다
JSON은 머신용. 사람은 1페이지 요약을 본다. 그리고 단순 룰 기반 권고 — "Top hotspot: X (change=15, coupling=45). 책임 분리·테스트 추가 우선." — 만으로도 다음 행동 가이드. AI 권고는 그 위의 사치.

### D. **결정론적 정량 분석이 AI보다 먼저**
hotspot Top N → AI review에 N=1로 시작. 만약 정량이 없었다면 AI는 "어느 파일?"부터 헤맸을 것. **AI는 우선순위가 결정된 후의 정성 평가에만 사용**. 토큰 비용 작고 결과 구체적.

### E. self-contained 단일 HTML이 빌드 파이프라인을 이긴다
대시보드 #6에서 React/Vue 검토했지만 1차 비대상으로 넘김. 결과: 4000라인 단일 HTML, D3 CDN 1줄, 빌드 0개, `open` 1번. 사용자는 빌드/배포 학습 없음. 1인 도구의 단일 HTML 패턴은 강력함.
